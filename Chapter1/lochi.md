## 1. 객체, 설계

### 무엇이 문제인가

모든 소프트웨어 모듈에는 세 가지 목적이 있다.

1. 실행 중에 제대로 동작하는 것: 모든 소프트웨어의 존재 이유.
2. 변경을 위해 존재하는 것: 간단한 작업만으로도 변경이 가능해야 함
3. 코드를 읽는 사람과 의사소통하는 것: 특별한 훈련 없이도 개발자가 쉽게 읽고 이해할 수 있어야 함.

예상을 빗나가는 코드

- 현실과 전혀 다른 방식으로 의사소통 하는 코드

변경에 취약한 코드

- 변경이 생기면 다른 여러 코드에도 변화가 생기게 됨
- “의존성”
- 객체 상의 의존성을 완전히 없애는 것은 정답은 아니다.
- 애플리케이션의 기능을 구현하는데 필요한 최소한의 의존성만 유지하고 불필요한 의존성은 제거하는 것

### 설계 개선하기

- 객체를 자율적인 존재로 만들자 ← 책임의 이동
  - 캡슐화로 코드를 숨기자
  - 인터페이스에만 의존하게 한다.
- 변경에 유리 ← 결합도를 낮춤 ← 자율적인 존재로 만들기 ← 책임의 이동 ← 의존성을 제거 ← 캡슐화

실세계와 객체지향 세계

- 실세계에서 상호작용하는 방식과 객체지향 세게에서 작동하는 방식은 다를 수 있음
  - 실세계: 소극장에 입장하기 위해서는 누군가 소극장의 문을 열고 입장을 허가해줘야 함.
  - 객체지향 세계: 티켓이 있는지 본인이 확인하고 소극장이 들여보냄
- 비록 실세계에서는 수동적인 존재여도, 객체지향 세계에서는 모든 것이 능동적이고 자율적인 존재임

### 객체지향 설계

설계

- 설계한 코드를 배치하는 것
- 동작 자체는 똑같을 수는 있어도, 코드의 배치는 달라질 수 있음

객체지향 설계

- 의존성을 효율적으로 통제해서, 요구사항 변경에 좀 더 수월하게 대응할 수 있는 가능성 높여줌
- 훌륭한 객체지향 설계란 협력하는 객체 사이의 의존성을 적절하게 관리하는 설계다.
